# Framework Agent Pickup: fw fabric context — LLM Context Enrichment

> Task: T-118 | Generated: 2026-03-12

## What You Need To Do

Add a new `fw fabric context` subcommand to the framework that generates a structured
architecture summary optimized for LLM consumption. This enables any project using the
framework to inject architectural context into LLM sessions automatically.

## OneDev Location

- Instance: `onedev.docker.ring20.geelenandcompany.com`
- Repo: `agentic-engineering-framework`
- Branch: create `feature/fabric-context-command`

## Problem

LLMs assisting on a project need architectural context to make informed decisions.
Currently, an LLM must run multiple `fw fabric` commands (overview, deps, impact)
to understand the system. Each query costs context tokens and requires the LLM to
synthesize results. A single command should produce a token-efficient structured
document suitable for injection into system prompts or CLAUDE.md.

## Evidence from 010-termlink

This project has 5 subsystems, 43 components, 80 edges. Understanding the architecture
requires running:
- `fw fabric overview` — subsystem summaries
- `fw fabric deps <path>` — per-component dependencies (43 times for full picture)
- `fw fabric drift` — health check

A single `fw fabric context` would produce all of this in one shot, structured for
LLM consumption.

## Proposed Implementation

### New file: `agents/fabric/fabric-context.sh` (or add as subcommand to `fabric.sh`)

The command should:
1. Read `subsystems.yaml` for high-level architecture
2. Read all component cards from `.fabric/components/`
3. Generate a structured output with:
   - **Subsystem map** — name, purpose, component count, dependencies
   - **Key interfaces** — public APIs, entry points, important types
   - **Dependency graph** — simplified (subsystem-level, not component-level)
   - **Component index** — path, purpose, type (one line per component)

### Output format options (decide during implementation):

**Option A: Markdown** — human-readable, injectable into CLAUDE.md
```markdown
## Architecture Context (auto-generated by fw fabric context)

### Subsystems
- **protocol** (5 components): Wire protocol — JSON-RPC + binary frames
- **session** (16 components): Session lifecycle — handlers, events, PTY
...

### Dependency Graph
protocol → session → hub
protocol → session → cli
session → hub → agent-mesh
...

### Components (43 total)
| Path | Type | Purpose |
|------|------|---------|
| crates/termlink-protocol/src/lib.rs | module | Protocol root... |
...
```

**Option B: YAML** — machine-parseable, can be post-processed
```yaml
generated: 2026-03-12T16:00:00Z
subsystems: [...]
graph: [...]
components: [...]
```

**Recommendation:** Markdown (Option A) — directly injectable, human-reviewable, no post-processing needed.

### Size target

< 2K tokens for a typical project (5-10 subsystems, 30-50 components). Larger projects
should have a `--compact` flag that omits the per-component index.

### Integration point

After implementation, the SessionStart hook or `/resume` skill could auto-inject the
output into context. But that's a follow-up — the command itself is the deliverable.

```
## Problem

LLMs need architectural context but must run multiple fw fabric commands to get it.

New `fw fabric context` subcommand that reads subsystems.yaml and component cards,

fw fabric context --compact    # Subsystems + graph only (no component index)
fw fabric context > .context/architecture.md  # Save for injection
```
