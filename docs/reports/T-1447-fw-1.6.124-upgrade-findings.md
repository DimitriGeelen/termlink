## Findings — classified

### A. Framework bugs (REPORT upstream — STEP 5: do NOT fix locally)

#### F-1 — `fw vendor` excludes `tests/` from copy → `fw test all` broken on consumer installs

`do_vendor()` in `bin/fw` (line ~245) hard-codes the includes list as `bin lib agents web docs .tasks/templates FRAMEWORK.md metrics.sh`. The `tests/` directory is never copied. Yet `bin/fw` test handler (line ~4885) runs `bats $FRAMEWORK_ROOT/tests/unit/` — guaranteed ERROR on every consumer install.

**Inconsistency:** `fw doctor` correctly recognises consumer mode and emits `SKIP Test infrastructure (consumer project — tests live in framework repo)`. `fw test all` does not.

**Fix options** (upstream choice):
- (a) Add `tests` to vendor includes list, OR
- (b) Have `fw test` short-circuit when `[ ! -d $FRAMEWORK_ROOT/tests/unit ]`, emitting the same SKIP message doctor does.

#### F-2 — `fw upgrade` "governance refresh" clobbers project-specific governance content

Step 1/10 ("Governance sections refreshed from framework template") replaced 3 lines of project-specific content on this run:

```
- 803c803  consumer-vendored-path rule shortened to generic "use bin/fw not fw"
- 991      "/agent-handoff" Quick Reference row deleted
- 992      "/check-arc"     Quick Reference row deleted
```

The merge logic preserves *some* project sections but does not preserve project-added rules INSIDE the governance section, nor project-added rows INSIDE the Quick Reference table. Backup is written to `CLAUDE.md.bak` so manual restore is possible — but the operator has to know to diff and restore.

**Fix:** section-aware merge that detects and preserves project additions inside governance regions, OR explicit `# project-specific` markers that the upgrader respects.

#### F-3 — Web tests assume framework-source project data

5 of 7 web failures (`test_project_page_lists_docs`, `test_project_doc_renders_markdown`, `test_patterns_cards_link_to_tasks`, `test_dashboard_has_system_health`, `test_404_for_nonexistent_task`) hard-code expectations that match the framework's own project files (e.g. doc `001-Vision`, dashboard string `System Health`, `/tasks/T-` link presence). Run against a consumer project they fail because the consumer doesn't ship those exact docs.

These tests should:
- Use `tmp_path` + monkeypatched PROJECT_ROOT to a fixture project with the expected content, OR
- Be marked framework-internal (skipped on consumer installs)

#### F-4 — `web/test_app.py::TestEmptyTaskFiles` monkeypatch leaks

`test_task_file_no_frontmatter` and `test_task_file_empty` monkeypatch `web.blueprints.tasks.PROJECT_ROOT = tmp_path` and assert the test task IDs (T-997, T-998) are NOT in the response. The response is 4 321 005 bytes — clearly the live project's task list — and the test IDs leak through. The monkeypatch isn't propagating to all task-listing code paths (template helper? cached config? alternate import?).

#### F-6 — Playwright `conftest.py` resolves `PROJECT_ROOT` from env var → wrong cwd on consumer installs

`tests/playwright/conftest.py:32` reads `os.environ.get('PROJECT_ROOT', ...)`. When `fw test playwright` runs on a consumer install, the framework has already set `PROJECT_ROOT=/opt/termlink` (the consumer root). The fixture then runs `subprocess.Popen(['python3', '-m', 'web.app', ...], cwd=project_root)` — which tries to import `web.app` from the consumer root, where `web/` does not exist (it's in FRAMEWORK_ROOT).

Result: every test fails fixture setup with `ModuleNotFoundError: No module named 'web'`. 444 ERRORs on first run. Applying `PROJECT_ROOT=$FRAMEWORK_ROOT` override drops it to **47 failed / 384 passed / 16 skipped**.

**Fix:** conftest should compute the framework root from `__file__` (already does, as fallback) and either:
- (a) ignore `PROJECT_ROOT` env var entirely for this specific subprocess `cwd`, OR
- (b) read a separate `FRAMEWORK_ROOT` env var (the framework sets one) and use that instead.

The remaining 47 failures concentrate in pages/APIs that depend on framework-source data (review/inception/tasks/verdict) — same family as F-3.

#### F-5 — `fw test all` does NOT include `lint`

Despite the `all` semantic, lint is omitted. Either include it, or document the omission in the help text. Operators reasonably expect "all" to mean all.
