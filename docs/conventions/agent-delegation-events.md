# Agent Delegation Event Schema Convention

Standard event topics and payload schemas for agent-to-agent task delegation via TermLink.

**Source:** T-012 inception research (`docs/reports/T-012-agent-to-agent-communication.md`)

## Event Topics

| Topic | Direction | Purpose |
|-------|-----------|---------|
| `task.delegate` | orchestrator → specialist | Assign work to a specialist agent |
| `task.accepted` | specialist → orchestrator | Acknowledge receipt and start of work |
| `task.completed` | specialist → orchestrator | Report successful completion with results |
| `task.failed` | specialist → orchestrator | Report failure with error details |

## Lifecycle

```
Orchestrator                          Specialist
    |                                     |
    |--- task.delegate -----------------→ |
    |                                     |-- (validates, starts work)
    |←-------------- task.accepted -------|
    |                                     |
    |                                     |-- (does work, writes results to disk)
    |                                     |
    |←-------------- task.completed ------|  (success)
    |        OR                           |
    |←-------------- task.failed ---------|  (failure)
```

## Payload Schemas

### task.delegate

Sent by the orchestrator to assign work to a specialist.

```json
{
  "request_id": "req-<pid>-<timestamp>",
  "task_id": "T-058",
  "action": "review",
  "scope": {
    "files": ["src/main.rs", "src/handler.rs"],
    "criteria": "security, error handling"
  },
  "reply_to": "orchestrator-session-name",
  "timeout_secs": 300
}
```

| Field | Required | Description |
|-------|----------|-------------|
| `request_id` | yes | Unique correlation ID. Format: `req-<pid>-<timestamp>`. Auto-generated by `termlink request`. |
| `task_id` | no | Framework task ID if the delegation maps to a tracked task. |
| `action` | yes | What the specialist should do (e.g., `review`, `test`, `build`, `analyze`). |
| `scope` | yes | Action-specific payload describing the work. Free-form object. |
| `reply_to` | yes | TermLink session name where the specialist should emit reply events. |
| `timeout_secs` | no | How long the orchestrator will wait before considering the delegation timed out. Default: 300. |

### task.accepted

Sent by the specialist to acknowledge receipt. Optional but recommended for long-running tasks.

```json
{
  "request_id": "req-001",
  "task_id": "T-058",
  "status": "accepted"
}
```

| Field | Required | Description |
|-------|----------|-------------|
| `request_id` | yes | Must match the `request_id` from `task.delegate`. |
| `task_id` | no | Echo back if provided in the delegation. |
| `status` | yes | Always `"accepted"`. |

### task.completed

Sent by the specialist when work is done successfully.

```json
{
  "request_id": "req-001",
  "task_id": "T-058",
  "status": "completed",
  "result_path": "docs/reviews/T-058-review.md",
  "summary": "3 issues found: 1 critical (SQL injection), 2 minor"
}
```

| Field | Required | Description |
|-------|----------|-------------|
| `request_id` | yes | Must match the `request_id` from `task.delegate`. |
| `task_id` | no | Echo back if provided. |
| `status` | yes | Always `"completed"`. |
| `result_path` | yes | Path to the result artifact on disk. Follows the "write to disk, return path" convention. |
| `summary` | yes | One-line summary of the result. Kept short to avoid context budget bloat on the orchestrator. |

### task.failed

Sent by the specialist when work cannot be completed.

```json
{
  "request_id": "req-001",
  "task_id": "T-058",
  "status": "failed",
  "error": "Could not access file: permission denied",
  "recoverable": true
}
```

| Field | Required | Description |
|-------|----------|-------------|
| `request_id` | yes | Must match the `request_id` from `task.delegate`. |
| `task_id` | no | Echo back if provided. |
| `status` | yes | Always `"failed"`. |
| `error` | yes | Human-readable error description. |
| `recoverable` | no | Whether the orchestrator could retry (e.g., transient failure vs. missing capability). Default: `false`. |

## Usage with TermLink Commands

### Orchestrator side

```bash
# One-shot: delegate and wait for reply
termlink request worker \
  --topic task.delegate \
  --payload '{"request_id":"req-001","action":"review","scope":{"files":["src/main.rs"]},"reply_to":"orchestrator"}' \
  --reply-topic task.completed \
  --timeout 300

# Manual: emit then watch
termlink emit worker task.delegate --payload '...'
termlink wait orchestrator --topic task.completed --timeout 300
```

### Specialist side

```bash
# Watch for incoming delegations
termlink watch --topic task.delegate

# Acknowledge
termlink emit orchestrator task.accepted --payload '{"request_id":"req-001","status":"accepted"}'

# Report completion
termlink emit orchestrator task.completed \
  --payload '{"request_id":"req-001","status":"completed","result_path":"docs/reviews/review.md","summary":"All clear"}'
```

### Spawning a specialist

```bash
# Start specialist in new terminal, wait for registration
termlink spawn --name reviewer --roles reviewer --wait -- \
  claude -p "You are a code reviewer. Watch for task.delegate events and review the specified files."
```

## Design Rules

1. **`reply_to` enables decoupling.** The specialist doesn't need to know the orchestrator's name at spawn time — it's in the delegation payload.
2. **`result_path` follows "write to disk, return path."** Full results go to files, not event payloads. This preserves context budget on both sides.
3. **`request_id` enables concurrent delegations.** An orchestrator can delegate to multiple specialists simultaneously and correlate replies.
4. **`task_id` bridges TermLink and the framework task system.** When present, the specialist should create or work under this framework task.
5. **Topics are hierarchical by convention.** `task.*` for delegation lifecycle. Future extensions: `session.*` for lifecycle, `health.*` for monitoring.
