# T-2456: receipt frontier aggregation must be monotonic by up_to not latest-by-ts

## Context

Round-13 adversarial review of the durable-delivery receipt path (`channel.receipts`,
T-1329/T-2286). The hub-side aggregation in `crates/termlink-hub/src/channel.rs`
(`ReceiptsWalk`, ~lines 1263-1272) keeps the **latest receipt per sender by `ts`**
(tie-broken by higher `up_to`). But a receipt's `up_to` means "received up to
offset N" — a delivery high-water mark that can only advance. Latest-by-ts lets a
recipient that posts a *later-timestamped* receipt with a *smaller* `up_to` move its
own advertised frontier **backwards**, which (a) is semantically wrong for a
delivery-confirmation frontier and (b) lets a recipient re-open `--await-ack`
re-posts for a message it already confirmed (bounded self-sabotage amplification,
LOW). Fix: aggregate by **max `up_to`** (tie-broken by newer `ts`) so the frontier
is monotonic. Behaviour-preserving for honest senders (whose `up_to` rises with
`ts`); only the pathological receding case changes. Origin: Q3 of the round-13
review (the CLEAN-but-LOW note); the HIGH/MED identity-binding siblings (cv_index
cv_key spoof, rate-limit bucket spoof) are filed separately as an inception.

## Acceptance Criteria

### Agent
- [x] `ReceiptsWalk` aggregation in `channel.rs` selects the entry with the **maximum
      `up_to`** per sender (tie-broken by newer `ts`), replacing latest-by-ts, so a
      receipt with a smaller `up_to` can never lower a sender's reported frontier.
- [x] A unit test posts, for one sender, `up_to=5 (ts=100)` then `up_to=3 (ts=200)`
      and asserts the aggregated frontier reports `up_to=5` (would be `3` under the
      old latest-by-ts logic).
- [x] Existing receipt/await-ack tests still pass (no regression for the honest
      monotonic case): `cargo test -p termlink-hub --lib receipt` green.
