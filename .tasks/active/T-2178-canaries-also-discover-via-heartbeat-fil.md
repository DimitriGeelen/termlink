# T-2178: /canaries: also discover via .heartbeat files — close the never-fired-canary visibility gap

## Context

`scripts/canary-status.sh` (T-2172) auto-discovers canaries via `.context/working/.*-canary.log` glob — so a healthy, never-fired canary is invisible to the operator. Just observed live with T-2175's substrate-preflight canary: heartbeat exists from the smoke run, but the `.log` is absent (cron has not yet fired, or no FAIL has occurred). `/canaries` reports only 3 canaries instead of 4. `substrate-cron-recipes.md` line 419-423 sample even shows the substrate-preflight-canary with `log=--`, implying it SHOULD render — the doc was aspirational. This task closes the gap by extending discovery to also glob `.*-canary.heartbeat` and synthesize the matching `.log` path for the renderer (classify() already gracefully handles log_size=0 / log_mtime=0 from absent files).

## Acceptance Criteria

### Agent
- [x] `scripts/canary-status.sh` discover_canaries() also globs `.*-canary.heartbeat`, deriving the corresponding `.log` path for the renderer. Result is `sort -u` deduped (no double-counting when both files exist).
- [x] Live smoke: `bash scripts/canary-status.sh` lists 4 canaries (was 3), with `substrate-preflight-canary  HEALTHY  hb=<recent>  log=--`.
- [x] Regression: existing 3 canaries (release-mirror, fleet-doorbell-mail, canary-aliveness) still render with their full metadata.
- [x] JSON envelope (`--json`) carries the new heartbeat-only canary as `{name, status, log_size:0, log_mtime:0, heartbeat_mtime:<int>}`.
- [x] STALE classification still works for heartbeat-only case (heartbeat older than threshold → STALE).
