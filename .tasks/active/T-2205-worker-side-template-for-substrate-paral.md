# T-2205: Worker-side template for substrate parallel-worker pattern (T-2204 follow-up)

## Context

Worker-side companion to T-2204's `scripts/orchestrator-backlog-drain.sh`. Completes the substrate consumer kit: an AEF integrator clones BOTH scripts and has a complete dry-run-first orchestrator + worker pair to drop on their host.

Worker polls for claims held by itself on the `work-queue` topic (via the substrate primitive #1 read surface — `channel claims-summary`), reads each work-unit envelope, renders the operator brief, and surfaces an ack/retry/skip decision. Mirrors the orchestrator's --dry-run default with a --live operator-prompt mode and an --auto-noop substrate-smoke-test mode that releases without doing the work.

## Acceptance Criteria

### Agent
- [x] Worker script `scripts/worker-backlog-drain.sh` exists and is executable
- [x] Three modes wired: `--dry-run` (default, no hub writes), `--live` (operator stdin prompt), `--auto-noop` (release without work — substrate smoke-test only)
- [x] Worker identity resolves via the same chain as orchestrator: `--worker-id` flag → `TERMLINK_AGENT_ID` env → `~/.termlink/be-reachable.state`
- [x] Reads `channel claims` for `$QUEUE_TOPIC`, filters to claims where `claimer == $WORKER_ID` (uses the LIST verb T-2037, NOT `claims-summary` which is aggregate-only — verified via end-to-end simulation with posted T-9999 work-unit)
- [x] Per held claim: prints offset, claim_id, ttl_remaining + task_id, classification, ac_count, dispatched_by, and the appropriate brief (closure-ready vs needs-work) — payload decoded from base64 `payload_b64` envelope field
- [x] `--once` flag for single-pass mode; default is polling loop with `--poll-interval` (default 15s)
- [x] SIGINT exits cleanly (loop trap on INT and TERM)
- [x] Recipe doc `docs/operations/substrate-orchestrator-recipe.md` references the worker script alongside the orchestrator in the in-tree consumer section (new "Worker-side companion" subsection)
